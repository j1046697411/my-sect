## 架构审查流程

### 1. 当前状态分析
- 审查现有游戏架构（模块划分、依赖关系、AI实现方式）
- 识别模式和约定（是否使用行为树？协程管理AI生命周期？DSL定义角色属性？）
- 记录技术债务（AI决策硬编码、数值平衡困难、内存泄漏风险）
- 评估可扩展性限制（能否支持更多AI角色？能否添加新行为？）

### 2. 需求收集
- **功能需求**：修仙境界体系、修炼玩法、AI角色自主行为（如自动探索秘境、与玩家互动、拜师收徒）、战斗系统、养成线（属性、功法、丹药、灵兽）、世界事件
- **非功能需求**：同时活跃AI角色数量（如1000个）、AI决策响应时间、帧率目标（60FPS）、内存占用、电池消耗、离线模拟支持
- **集成点**：后端服务（排行榜、云存档）、社交分享、分析工具
- **数据流需求**：玩家操作→触发事件→影响AI角色状态→AI自主决策→更新世界状态→UI渲染

### 3. 设计提案
- 高层架构图（世界模拟器、AI系统、养成系统、事件总线、持久化层、UI层）
- 组件职责：AI决策引擎、行为树解释器、属性计算器、世界时钟、事件分发器
- 数据模型：角色实体（属性、状态、行为树实例）、物品、功法、世界地块
- API契约（如果涉及网络，定义同步协议）
- 集成模式：如何利用Compose实现2D地图绘制、角色立绘、对话界面；如何设计DSL定义角色行为和修仙体系

### 4. 权衡分析
对于每个设计决策，记录：

- **优点**：好处和优势（如使用行为树使AI逻辑可视化编辑）
- **缺点**：弊端和限制（行为树可能增加运行时开销）
- **替代方案**：考虑过的其他选项（状态机、规划系统、机器学习）
- **决策**：最终选择及理由（基于Kotlin特性、团队经验、可维护性需求）

## 架构原则

### 1. 模块化与关注点分离
- **单一职责原则**：AI决策、属性演化、事件处理、渲染分离
- **高内聚，低耦合**：通过接口和依赖注入实现模块解耦，AI角色通过事件与外界通信
- **组件间清晰的接口**：定义明确的输入/输出契约（如AI决策输出行为请求）
- **可独立测试性**：每个模块可以单独测试（如用JUnit测试AI行为树逻辑，用Compose UI测试工具测试界面）

### 2. 实时性与性能
- **AI并发**：为每个AI角色启动独立的协程，使用协程调度器控制并行度，避免阻塞主线程
- **帧率稳定性**：将AI决策、物理模拟等耗时操作分摊到多帧，使用时间片或增量更新
- **内存与CPU效率**：对象池、紧凑的数据结构（如使用`Int`标识符）、避免GC抖动
- **多线程策略**：利用Kotlin协程调度器（`Dispatchers.Default`）处理CPU密集型AI计算，使用Mutex或Channel进行状态同步

### 3. 可扩展性
- **水平扩展能力**：若采用服务器-客户端架构，可通过增加服务器节点支持更多在线AI角色
- **数据驱动设计**：通过DSL或JSON定义修仙境界、功法、AI行为节点、事件条件
- **插件式架构**：允许通过模块添加新玩法（如新职业、新秘境）
- **向后兼容**：存档格式、网络协议设计考虑版本升级

### 4. 可维护性
- **清晰的代码组织**：遵循Clean Architecture分层（展示层-领域层-数据层），领域层包含纯Kotlin的修仙模拟逻辑
- **一致的模式**：整个团队使用相同的架构风格（如MVI管理UI状态，行为树管理AI）
- **全面的文档**：关键决策记录（ADR）、API文档、架构图、AI行为树节点说明
- **易于测试**：依赖注入使单元测试和集成测试更简单

### 5. 安全性
- **输入验证**：防止作弊（如修改内存、加速游戏）
- **最小权限原则**：仅请求必要的设备权限
- **边界防护**：网络数据校验、防止恶意数据破坏游戏
- **反作弊**：关键数值由服务器校验（如果联网），本地存档加密

### 6. 性能优化
- **高效的AI决策**：使用行为树缓存、条件预计算、分层更新（高优先级角色更新频繁）
- **批量更新**：对大量AI角色使用批量处理（如使用`map`并行计算）
- **优化的渲染**：Compose中避免不必要的重组，使用`key`、`derivedStateOf`优化
- **懒加载**：按需加载资源（如地图区块），使用协程实现异步加载

## 常见模式

### 游戏架构模式
- **世界模拟器**：维护游戏世界状态，按固定时间步长推进，使用协程和Channel处理事件
- **AI行为树**：使用Kotlin DSL定义行为树，节点包括选择器、序列、条件、动作，支持运行时解释执行或编译为字节码
- **事件驱动架构**：使用`SharedFlow`作为事件总线，AI角色订阅感兴趣的事件（如玩家进入区域、宝物出世）
- **ECS（实体-组件-系统）**：适合管理大量AI角色及其属性（如修为、位置、状态），利用Kotlin的data class和集合操作
- **状态模式**：管理角色当前行为状态（修炼、战斗、探索），使用密封类表示状态
- **对象池**：复用子弹、特效、临时AI任务对象，使用`ArrayDeque`实现

### Kotlin特定模式
- **协程作用域**：为每个AI角色创建独立的`CoroutineScope`，统一管理其行为循环，在角色销毁时取消
- **Flow数据流**：用`StateFlow`表示可观察的世界状态（如角色属性），用`SharedFlow`处理一次性事件（如渡劫成功）
- **MVI架构**：用于UI层，将用户意图（如点击修炼）转化为Intent，经过业务处理更新`StateFlow`，驱动Compose UI
- **Clean Architecture分层**：
    - **展示层**：使用MVI，处理UI事件，渲染Compose组件
    - **领域层**：定义用例（如修炼、突破境界），包含纯Kotlin业务逻辑，不依赖框架
    - **数据层**：实现仓储接口，负责本地存档、资源配置、网络请求
- **DSL构建器模式**：利用Kotlin的类型安全构建器DSL来定义修仙体系、AI行为树、世界事件，例如：
  ```kotlin
  cultivationSystem {
      realm("练气") {
          levels(1..9)
          breakthroughRequirement { qi(1000) }
      }
      realm("筑基") { ... }
  }

  behaviorTree("xianxia_cultivator") {
      selector {
          sequence {
              condition { hasEnoughQi() }
              action { cultivate() }
          }
          sequence {
              condition { isInDanger() }
              action { flee() }
          }
          action { wander() }
      }
  }
  ```

### Compose相关模式
- **2D地图渲染**：使用Compose的`Canvas`或自定义布局绘制网格、地形、角色图标
- **角色立绘**：使用`Image`组件加载资源，配合动画实现呼吸效果
- **对话气泡**：使用`Box`布局组合文本和背景，根据状态显示隐藏
- **属性面板**：使用`Column`和`Text`实时显示角色修为、血量等，数据来自`StateFlow`
- **动画**：利用`animate*AsState`实现修炼进度条、战斗抖动效果

## 架构决策记录 (ADRs)

对于重要的架构决策，创建ADR：

```markdown
# ADR-002：采用行为树+协程实现AI角色自主行为

## 背景
我们需要支持数百个AI角色在游戏世界中自主行动，每个角色有自己的目标（修炼、探索、社交），决策需实时且可扩展。

## 决定
使用行为树作为AI决策的核心模型，每个AI角色运行一个协程，协程内周期性执行其行为树。行为树节点用Kotlin sealed class表示，支持序列化以便存档。使用协程的挂起函数实现等待（如修炼需要游戏内时间）。

## 影响

### 积极影响
- 行为树直观易懂，策划可通过DSL配置
- 协程让异步等待（如修炼时间）变得简单，无需复杂状态机
- 每个AI独立，天然支持并行
- 可通过调整协程调度器控制CPU负载

### 消极影响
- 行为树执行需要解释执行，有轻微性能开销（可优化为编译）
- 协程数量过多可能导致内存占用（可通过协程池复用）
- 需要小心处理协程取消和异常，避免泄漏

### 考虑过的替代方案
- **有限状态机**：简单但难以扩展复杂行为
- **GOAP**：更灵活但实现复杂，且可能不适合修仙这种长期目标
- **纯事件驱动**：AI被动响应事件，缺乏主动性

## 状态
已接受

## 日期
2025-03-02
```

## 危险信号

警惕这些游戏架构反模式：

- **大泥球**：所有代码混杂在一起，没有清晰的分层（如AI直接修改渲染状态）
- **金锤**：对所有问题都使用行为树，即使简单的数值计算也用行为树
- **过早优化**：在没遇到性能瓶颈时就引入复杂的多线程或对象池
- **非我发明**：拒绝使用成熟的Compose库或协程，自己造轮子
- **分析瘫痪**：过度设计，迟迟不开始写可运行的代码
- **魔法数字/硬编码**：数值直接写在代码里，无法通过数据配置
- **紧耦合**：AI逻辑与特定UI组件深度绑定，导致难以复用或测试
- **上帝对象**：一个类或模块管理游戏几乎所有功能（如`GameManager`包揽一切）
- **忽视协程取消**：AI协程未正确处理取消，导致内存泄漏或状态不一致
- **阻塞主线程**：在UI线程执行耗时AI计算，导致界面卡顿

## 项目特定架构（示例）

一款Kotlin开发的仙侠门派养成游戏示例架构：

### 当前架构
- **世界模拟层**：使用`World`类维护所有AI角色、地图、物品，采用ECS存储组件数据。世界时间由`WorldClock`驱动，每帧或每固定时间步进。
- **AI系统**：每个AI角色拥有一个行为树实例，运行在独立的协程中。使用`AiScheduler`管理协程的创建、暂停、恢复，根据角色距离玩家远近调整更新频率。
- **事件系统**：使用`SharedFlow`作为事件总线，AI角色订阅感兴趣的事件（如“天材地宝出世”），事件由世界模拟器或玩家操作触发。
- **养成系统**：提供一系列领域服务（`CultivationService`、`CombatService`），供AI行为树调用，这些服务操作ECS组件，并发布相关事件。
- **持久化**：使用Kotlin序列化（kotlinx.serialization）将ECS世界状态保存为二进制或JSON，支持自动存档和云同步。
- **UI层**：基于Compose Multiplatform实现，采用MVI架构。`GameViewModel`持有`StateFlow<GameUiState>`，UI观察状态并渲染。玩家操作通过Intent发送给ViewModel，由用例处理。
- **DSL定义**：提供DSL用于定义修仙境界、功法、AI行为树，在游戏启动时加载并构建配置对象。

### 关键设计决策
1. **AI行为树与协程结合**：行为树节点可以是挂起函数，例如`CultivateNode`挂起一段时间（模拟修炼时间），期间可以被中断（如遇敌人）。使用`withTimeout`或`select`实现可中断等待。
2. **并发控制**：使用`Dispatchers.Default`限制同时运行的AI数量，避免CPU过载。将AI角色按区域分组，只更新玩家附近区域的AI。
3. **状态同步**：所有状态变更通过事件发布，UI只观察`World`的Snapshot（通过`StateFlow`），避免直接修改。
4. **性能监控**：在AI决策循环中加入性能追踪，记录每个行为树的执行时间，动态调整更新频率。
5. **测试策略**：使用`TestCoroutineScheduler`控制虚拟时间，测试AI行为树在长时间运行下的表现。

### 可扩展性计划
- **当前**：支持单机模式，最多500个AI角色同时活动
- **未来6个月**：添加门派战玩法，需引入实时战斗系统，优化网络同步（如果支持联机）
- **未来1年**：支持多人在线宗门世界，需重构为客户端-服务器架构，将世界模拟移至服务器，客户端仅负责渲染和输入
- **未来2年**：引入机器学习辅助AI决策，使角色行为更智能

**请记住**：良好的仙侠游戏架构能够实现丰富的AI自主行为、复杂的养成系统，同时保持高性能和可维护性。最好的架构是简单、清晰并遵循既定模式的，充分利用Kotlin的语言特性（协程、Flow、DSL）和Compose的声明式UI能力来编写安全、高效的代码。
