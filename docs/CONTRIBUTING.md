# 文档贡献指南

本文档说明如何为项目文档做出贡献和维护。

---

## 1. 什么需要文档 (What Needs Documentation)

以下情况**必须**更新文档：

### API 变更
- 新增/修改/删除公共 API
- 函数签名变更（参数、返回值）
- 接口或抽象类变更
- 弃用通知

### 新功能
- 用户可见的功能
- 配置选项变更
- 新的工作流程或命令

### 架构决策
- 技术栈变更
- 目录结构重组
- 依赖管理策略
- 性能优化方案

### 行为变更
- 默认值变更
- 错误处理逻辑
- 兼容性破坏

---

## 2. 文档标准 (Documentation Standards)

### KDoc - 公共 API

所有公共 API 必须包含 KDoc：

```kotlin
/**
 * 简短描述（一句话）
 *
 * 详细描述（可选，说明用途、边界条件）
 *
 * @param paramName 参数说明
 * @return 返回值说明
 * @throws ExceptionType 异常条件
 */
fun publicApi(paramName: Type): ReturnType
```

**规则**：
- `public`/`protected` 成员必须有 KDoc
- `internal` 建议有 KDoc
- `private` 可选 KDoc（复杂逻辑时推荐）

### ADR - 架构决策记录

重大决策使用 ADR 格式，位于 `docs/architecture/adr/NNNN-title.md`：

```markdown
# ADR NNNN: 决策标题

## 状态
[提案 | 已接受 | 已弃用 | 已取代]

## 背景
为什么要做这个决策

## 决策
我们决定...

## 后果
- 正面影响
- 负面影响
- 需要跟进的事项
```

### 指南 - 工作流程

操作指南位于 `docs/guides/`：

- 使用步骤式格式（1. 2. 3.）
- 包含命令示例
- 说明预期输出
- 列出常见问题

### README 章节

根目录 `README.md` 包含：
- 项目简介（1-2 段）
- 快速开始（5 分钟内完成）
- 核心功能列表
- 相关链接

## 3. 更新流程 (Update Process)

### 何时更新

| 触发条件 | 更新内容 | 时机 |
|---------|---------|------|
| 新功能 PR | README + API 文档 | 与代码同 PR |
| API 变更 | KDoc + 变更日志 | 与代码同 PR |
| 架构决策 | ADR | 实施前 |
| Bug 修复 | 相关指南（如影响用户） | 修复后 |
| 版本发布 | 变更日志 + 迁移指南 | 发布时 |

### 如何更新

1. **确定范围**
   - 哪些文件受影响
   - 哪些用户受影响（开发者/终端用户）

2. **选择文档类型**
   - API 变更 → KDoc
   - 决策 → ADR
   - 流程 → Guide

3. **编写内容**
   - 使用清晰简洁的语言
   - 提供代码示例（如适用）
   - 包含验证步骤

4. **审查**
   - 运行文档检查清单（见下文）
   - 请求团队成员审查

5. **合并**
   - 文档与代码原子提交
   - 提交信息包含 `[docs]` 前缀

## 4. Guardrails (限制)

### 智能体可以做什么

- ✅ 根据代码变更建议文档更新
- ✅ 生成 KDoc 模板
- ✅ 检查文档覆盖率
- ✅ 格式化文档（拼写、语法）
- ✅ 提取代码注释到文档

### 智能体不可以做什么

- ❌ **不得**编造 API 行为（必须基于实际代码）
- ❌ **不得**修改技术决策（需人类确认）
- ❌ **不得**删除现有文档（只能建议弃用）
- ❌ **不得**创建未实现功能的文档
- ❌ **不得**覆盖人工编写的指南（只能建议更新）

### 人工审查必需

以下情况**必须**人工审查：
- ADR 创建或变更
- 公共 API 文档首次编写
- 弃用通知
- 迁移指南

---

## 5. 文档质量检查清单

提交前验证：

- [ ] **准确性**：文档与实际代码行为一致
- [ ] **完整性**：覆盖所有公共 API 和用例
- [ ] **清晰度**：无歧义，术语一致
- [ ] **简洁性**：无冗余，每段有明确目的
- [ ] **可搜索**：使用标准术语，包含关键词
- [ ] **示例正确**：代码示例可运行/可验证
- [ ] **链接有效**：所有内部/外部链接可访问
- [ ] **格式一致**：遵循项目文档风格
- [ ] **版本标注**：标注适用版本（如适用）
- [ ] **无障碍**：图片有 alt 文本，颜色对比度足够
